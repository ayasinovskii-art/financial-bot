# Автономный ночной конвейер разработки FinanceBot

**Дата:** 2026-06-08
**Статус:** проект (design), одобрен к написанию плана реализации

## Цель

Настроить автономную ночную разработку FinanceBot силами набора role-based AI-агентов,
работающих как последовательный конвейер: от идеи до готового pull request, который утром
ревьюит и мёржит человек, после чего изменения автоматически выкатываются в прод. Решение
должно укладываться в подписку **Claude Pro** (без API-счетов) и не давать угнать учётную
запись пользователя.

## Принятые ограничения

- **Бюджет:** в рамках Pro, без pay-as-you-go. Авторизация через `CLAUDE_CODE_OAUTH_TOKEN`
  (получается командой `claude setup-token`). `ANTHROPIC_API_KEY` **не задаётся** — иначе
  запросы уходят на платный путь.
- **Среда исполнения:** GitHub Actions. Почасовой `cron`-пульс с оппортунистическим повтором
  против 5-часового окна сброса лимитов Pro (точный «time-to-reset» на OAuth-пути недоступен).
- **Состояние вне сессии:** весь прогресс хранится в GitHub (issue + ветка + файл состояния),
  каждый шаг возобновляем с холодного старта.
- **Безопасность:** только эфемерный `GITHUB_TOKEN` (без личного PAT), branch protection на
  `master`, бот не имеет доступа к `.github/workflows/**`, third-party actions пиннуются на
  commit SHA.
- **Темп:** одна задача за ночь, ровно одна активная задача в каждый момент.

## Оркестрация (вариант B)

Один workflow-файл, несколько job'ов по стадиям. За один тик отрабатывает только тот job,
чья стадия совпадает с текущей меткой активного issue (`if:`-гейт по метке). Это даёт
**per-role изоляцию прав** (`permissions` и набор секретов у каждого job свои) при том, что
вся конфигурация в одном файле и естественно управляется почасовым cron'ом.

Рассмотренные и отклонённые альтернативы:
- **A — один job-оркестратор с суб-агентами:** проще по проводке, но нельзя сузить права под
  роль (job держит объединение всех прав) — слабее по безопасности.
- **C — отдельный workflow на роль, триггер от смены меток:** максимально «конвейерно», но
  больше движущихся частей и хуже ложится на пейсинг под 5-часовое окно.

## Модель состояния (state machine)

Состояние живёт в GitHub, вне сессии Claude:

- **Issue = тикет задачи.** Одна активная задача за ночь. Текущая стадия — это метка на issue:
  `stage:analysis` → `stage:planning` → `stage:coding` → `stage:qa` →
  `stage:security` → `stage:review` → `stage:done`, плюс `stage:blocked`. На issue всегда
  ровно одна метка `stage:*`; роль, завершая шаг, снимает свою и ставит следующую.
- **Тело issue = машинное состояние.** В теле issue поддерживается fenced ```json-блок (маркеры
  `<!-- pipeline-state -->`) с указателем стадии, выбранным вариантом реализации, планом-чек-листом
  и счётчиками итераций возврата. Правится через `issues: write` — то есть доступно и read-only-ролям
  (qa/security/reviewer), которым нужно инкрементить счётчики и переключать стадию. Код в репо они
  не трогают.
- **Ветка `bot/issue-<N>`** держит **только код**. Создаётся coder'ом на стадии `coding`, туда же
  он возвращается после QA/security/review. Никакого состояния конвейера в ветке нет.
- **Комментарии issue** — человекочитаемый журнал: каждая роль постит итог своего шага.

**Диспетчер тика.** Каждый почасовой запуск начинается с лёгкого шага-диспетчера (`gh`/скрипт,
без вызова Claude), который определяет, что делать:
1. Ищет открытый issue с меткой `stage:*`, кроме `done`/`blocked` → это активная задача, текущая
   стадия = её метка. Запускается соответствующий job.
2. Активной задачи нет, но есть issue с меткой `bot:queue` (или включён режим «менеджер придумывает
   сам») → стадия = `manager`.
3. Иначе (нечего делать) → idle-exit за секунды, без вызова Claude.

**Один тик = одна стадия.** Диспетчер выбирает стадию, запускается только соответствующий job, тот
делает свой шаг, обновляет json-блок в теле issue / переклеивает метку и выходит.

**Инвариант:** активная задача всегда ровно одна. Менеджер не заводит новую, пока текущая не
`done`/`blocked`. Это держит расход квоты предсказуемым.

## Роли

Семь ролей, каждая — отдельный job со своими `permissions` и промптом.

| Job | Стадия (in→out) | Модель | Права | Обязанности |
|---|---|---|---|---|
| **manager** | (нет) → `analysis` | **Opus** | `issues: write` | Берёт открытый issue с меткой `bot:queue`; если очередь пуста — придумывает идею (из `task.md`/`task2.md` + бэклога) и заводит issue с json-блоком состояния. Ставит `stage:analysis`. |
| **analyst** | `analysis` → `planning` | **Opus** | `contents: read`, `issues: write` | Предлагает 2–3 варианта реализации, выбирает лучший с обоснованием, пишет выбор в json-блок тела issue + коммент. |
| **techlead** | `planning` → `coding` | Sonnet | `contents: read`, `issues: write` | Декомпозирует выбранный вариант в чек-лист шагов + критерии приёмки. Кладёт план в json-блок тела issue. |
| **coder** | `coding` → `qa` | Sonnet | `contents: write`, `pull-requests: write`, `issues: write` | Создаёт/переиспользует `bot/issue-<N>`, реализует по плану, гоняет `dotnet build` + `dotnet test` локально, открывает/обновляет PR. **Единственная роль с правом записи в репо.** |
| **qa** | `qa` → `security` \| `coding` | Sonnet | `contents: read`, `issues: write` | Проверяет сборку, тесты, критерии приёмки и **покрытие**. Всё зелёное → `security`; иначе возврат в `coding`. |
| **security** | `security` → `review` \| `coding` | Sonnet | `contents: read`, `issues: write` | Аудит на инъекции/секреты/небезопасные паттерны. Чисто → `review`; иначе возврат в `coding`. |
| **reviewer** | `review` → `done` \| `coding` | **Opus** | `contents: read`, `pull-requests: write` | Ревью кода/архитектуры против `CLAUDE.md`. Апрув → коммент-апрув на PR + `stage:done`; замечания → возврат в `coding`. |

**Ключевой принцип безопасности:** запись в репозиторий есть только у `coder`. QA, security,
reviewer — строго read-only по `contents`; влияют на поток только через метки и `state.json`.
Ни у одной роли нет доступа к `.github/workflows/**`. PR **никто из ботов не мёржит** —
финальный мёрж в `master` только человек.

### Детали QA: покрытие

- QA собирает покрытие через `dotnet test --collect:"XPlat Code Coverage"` (coverlet).
- Критерий возврата в `coding`: покрытие нового/изменённого кода ниже порога (дефолт **70%**)
  **или** общий процент по solution упал относительно `master`.
- Вердикт QA = сборка ✅ + тесты ✅ + критерии приёмки ✅ + покрытие ✅.
- Если coverlet-коллектор ещё не подключён к тест-проектам — его настройка является
  первым («hello world») тикетом конвейера.

## Циклы возврата и потолки итераций

QA, security и reviewer могут вернуть задачу в `coding`. Счётчики живут в json-блоке тела issue:

```json
{
  "qa_iterations": 0,
  "sec_iterations": 0,
  "review_iterations": 0,
  "total_coding_passes": 0
}
```

Потолки (дефолты, подкручиваются правкой workflow):
- `qa_iterations ≤ 2`, `sec_iterations ≤ 2`, `review_iterations ≤ 2`
- `total_coding_passes ≤ 5` — общий предохранитель.

При превышении любого потолка задача уходит в `stage:blocked`: бот пишет коммент «застрял после
N итераций», добавляет метку `needs-human` и больше её не трогает.

## Поведение при исчерпании квоты

Оппортунистический повтор (точного «time-to-reset» на OAuth-пути нет):
- Job ловит ошибку лимита (429 / usage exhausted), пишет в лог `quota-exhausted`, завершается
  **успешно (exit 0)** и **не меняет метку**.
- Следующий почасовой тик повторит ту же стадию. Когда окно сбросится — шаг пройдёт.
- Следствие: каждый шаг должен быть **идемпотентным/возобновляемым** — coder переиспользует
  ветку и PR (не плодит новые), analyst/techlead перезаписывают свою секцию json-блока в теле issue.

**Пейсинг под Pro.** Один тик = одна стадия → за ночь (~8–10 тиков) реально пройти полный
конвейер по одной задаче даже с парой возвратов. Менеджер не заводит новую задачу, пока текущая
не `done`/`blocked`.

**Idle-тик дешёвый.** Если активной задачи нет и очередь пуста — workflow выходит за секунды
(cron ≈ 720 запусков/мес, бережём бесплатные минуты Actions).

## Безопасность Actions

**Токен Claude:**
- `CLAUDE_CODE_OAUTH_TOKEN` хранится в Encrypted Secrets репозитория; **периодическая ротация**
  вручную раз в N недель.
- `ANTHROPIC_API_KEY` не задаётся (иначе платный путь).
- Токен никогда не пишется в логи/артефакты/PR — явное правило в промптах + secret-masking
  Actions (включён по умолчанию).

**GitHub-сторона:**
- Только эфемерный `GITHUB_TOKEN`, никакого личного PAT. По умолчанию `permissions: {}` на
  workflow, расширяем точечно в каждом job (см. таблицу ролей).
- **Branch protection на `master`:** PR обязателен, мёрж только человеком, **required status
  check = CI** (см. раздел «CI и деплой»). Бот физически не может пушить в `master`, и PR нельзя
  смержить, пока CI-сборка/тесты не зелёные.
- **Агент не трогает `.github/workflows/**`:** (а) промпт coder'а запрещает менять воркфлоу;
  (б) `GITHUB_TOKEN` без `workflow` scope; (в) CODEOWNERS на `.github/**` = владелец репо.
- **Third-party actions пиннуются на полный commit SHA** (`@<40-hex>`), не на теги.
  `claude-code-action` — тоже по SHA.
- **Запрещён `pull_request_target`** и любые триггеры, дающие секреты форк-PR. Триггеры:
  конвейер — `schedule` (cron) + ручной `workflow_dispatch`; CI — `pull_request`; деплой —
  `push` в `master`. CI на `pull_request` бежит с read-only токеном и без секретов.
- **Self-hosted runner — только для деплоя и только на `push: master`.** Деплой-workflow помечен
  отдельным `runs-on: [self-hosted, financebot-deploy]` и срабатывает исключительно на пуш в
  `master` (то есть после ручного мержа человеком). Runner **никогда** не исполняет код из
  bot-PR'ов (CI/конвейер бегут на GitHub-hosted runner'ах). Это исключает запуск непроверенного
  кода на проде.
- **Issue-текст = недоверенный ввод** (prompt-injection): промпты оборачивают тело issue как
  данные, не как инструкции; менеджер/аналитик не выполняют команды из тела issue.
- **Secret scanning + push protection** включены на репо.
- **Concurrency-guard:** `concurrency: bot-pipeline`, `cancel-in-progress: false` — не дать двум
  тикам пересечься и задвоить расход/коммиты.

## Тестирование и наблюдаемость конвейера

**Наблюдаемость (для человека утром):**
- **Issue = единый журнал.** Каждая роль постит коммент с итогом шага: что сделала, вердикт,
  ссылки. Лента читается как «дневник ночи».
- **Метка стадии** показывает, где задача сейчас.
- **PR** — главный артефакт: diff + статусы `dotnet build`/`dotnet test`, ждёт мёржа человеком.
- **Сводный коммент менеджера** в конце прогона (или при `blocked`).
- Логи Actions — для глубокой отладки.

**Обкатка до боевого автозапуска:**
- **Поэтапная обкатка через `workflow_dispatch`** со входом `stage` — запуск одной роли вручную
  на тестовом issue, прежде чем включать cron.
- **`dry-run`-флаг** у coder: первый прогон без открытия PR (только лог «что бы сделал»).
- **Тестовый issue** с заведомо мелкой задачей (настроить coverlet + базовый порог) как
  end-to-end smoke-тест всего конвейера.
- **Cron включаем последним**, когда ручные прогоны всех стадий зелёные.

**Гейты качества кода — переиспользуем существующие:** `dotnet build FinanceBot.sln` (с
`TreatWarningsAsErrors`) и `dotnet test` гоняет coder локально перед PR, qa — повторно как
независимая проверка. Это объективная правда для QA-возврата.

## CI и деплой (после мержа)

Конвейер заканчивается на PR; дальше — независимый CI-гейт, ручной мёрж человеком и
авто-деплой.

**CI-гейт на PR (объективная правда, без квоты Claude).**
- Отдельный workflow `on: pull_request` (GitHub-hosted runner): `dotnet build FinanceBot.sln`
  (с `TreatWarningsAsErrors`) + `dotnet test` + сбор покрытия.
- Повешен как **required status check** в branch protection — PR нельзя смержить, пока CI не
  зелёный. Это страхует от случая, когда AI-роль (coder/qa) наврала в комментарии, что «всё
  прошло»: проверку делает не агент, а инфраструктура.
- Бежит с read-only токеном, без доступа к секретам конвейера/деплоя.

**Авто-деплой через self-hosted runner (вариант B).**
- На хосте бота установлен self-hosted GitHub runner с меткой `financebot-deploy`.
- Workflow `on: push: master` запускает на нём текущую процедуру редеплоя: полный rebuild +
  force-recreate с локальным `--env-file .env`.
- **Никаких деплой-секретов в GitHub** (ни SSH-ключа, ни доступа к серверу) — runner уже на
  хосте, `.env` лежит локально на сервере. Это держит атаку-поверхность минимальной.
- Триггер только `push: master` → деплой случается **только после ручного мержа человеком**.

**Миграции и безопасность данных при деплое.**
- EF-миграции применяются на старте (`DatabaseMigrationService`). Перед force-recreate
  деплой-скрипт делает **бэкап Postgres** (`pg_dump` в датированный файл), чтобы откатить данные,
  если миграция окажется деструктивной.
- **Rollback:** предыдущий docker-образ сохраняется (тег `:previous`); при неуспешном старте
  нового — быстрый возврат на предыдущий образ + восстановление из бэкапа при необходимости.
- Health-check после деплоя: если контейнер не поднялся/health-endpoint красный за N секунд —
  деплой считается провалившимся, выполняется rollback.

**Уборка веток.**
- В настройках репозитория включено **авто-удаление ветки при мерже PR** → `bot/issue-<N>`
  не накапливаются.
- Ветки задач в `stage:blocked` чистятся вручную при разборе.

**Уведомление утром.**
- Базово хватает GitHub-нотификаций о новом PR.
- Опционально (на потом): по завершении прогона бот шлёт короткий пинг в Telegram (через
  существующий `ITelegramBot`) — «PR #N готов / задача застряла». Не входит в первую итерацию.

## Открытые вопросы / на потом

- Точные пороги (потолки итераций, % покрытия) подкручиваются правкой workflow по факту.
- Расписание cron и «ночное окно» уточняются на этапе плана реализации.
- Сравнение покрытия с `master` требует базовой метрики: как и где хранить baseline покрытия
  master (артефакт, badge, отдельный замер в начале coding-стадии) — решается в плане реализации.
- Подтвердить, что используемая версия `claude-code-action` принимает `CLAUDE_CODE_OAUTH_TOKEN`
  и корректно работает на подписочной квоте без `ANTHROPIC_API_KEY`.
- Установка и регистрация self-hosted runner на хосте бота (метка `financebot-deploy`,
  запуск как сервис, ограничение прав пользователя runner'а) — детализируется в плане.
- Точная процедура деплой-скрипта (бэкап `pg_dump`, тег `:previous`, health-check, rollback)
  оформляется из текущих ручных шагов redeploy на этапе плана.
